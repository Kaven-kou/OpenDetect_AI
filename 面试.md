# OpenDetect_AI 项目架构与面试详解

项目基于 **LangGraph Supervisor 模式** 构建，涵盖多智能体工作流、RAG、MCP 协议、短期/长期记忆、SSE 流式推送等完整技术栈。

---

## 一、项目定位与核心价值

**目标**：为 AI 领域研究者提供一站式论文检索、入库、问答、综述生成的智能研究辅助系统。

**核心功能**：

| 功能 | 说明 | 使用场景 |
|------|------|----------|
| **智能搜索** | 模糊描述→精确论文映射，支持上下文指代词理解 | "帮我找首次提出 ViT 的论文" |
| **多方式入库** | 搜索/arxiv链接/本地PDF，自动去重，付费墙论文自动识别放弃 | 三种方式无缝衔接 |
| **RAG 问答** | 基于入库论文回答，标注来源，支持指代词追问 | "它相比 CNN 有什么优势？" |
| **综述生成** | 结构化综述 + 方法对比表，适合写 Related Work | 自动生成 Survey |
| **短期记忆** | 滑动窗口保留最近 4 轮对话 | 多轮连贯追问 |
| **长期记忆** | 跨会话持久化用户研究偏好 | 下次打开自动了解你的方向 |
| **实时进度** | SSE 流式推送各 Agent 执行进度 | 前端思考面板 |

## 二、整体架构

### 2.1 架构图

```
用户输入
    ↓
┌───────────────────────────────────────────────────┐
│            Supervisor Agent（调度中枢）             │
│  输入：长期记忆（用户偏好）+ 短期记忆（4轮上下文）  │
│  + 当前状态（搜索数/入库数/是否已回答）             │
│  输出：next = search / ingest / rag / report / FINISH │
└────────────────────┬──────────────────────────────┘
                     ↓ 条件路由
     ┌───────┬───────┼───────┬───────┐
     ↓       ↓       ↓       ↓       ↓
  Search  Ingest   RAG   Report  FINISH
  Agent   Agent   Agent  Agent
     │       │       │       │
     ↓       ↓       ↓       ↓
  OpenAlex Chroma DeepSeek DeepSeek
  优先，   向量库   LLM      LLM
  ArXiv补充
     │       │       │       │
     └───────┘       └───────┘
     回到 Supervisor      END
          ↓（FINISH）
         END → 后台线程异步提取用户偏好 → SQLite
```

### 2.2 LangGraph 图结构

```python
builder = StateGraph(AgentState)
builder.add_node("supervisor", supervisor_node)
builder.add_node("search",     search_node)
builder.add_node("ingest",     ingest_node)
builder.add_node("rag",        rag_node)
builder.add_node("report",     report_node)

builder.set_entry_point("supervisor")

# 条件路由：supervisor 决定去哪
builder.add_conditional_edges("supervisor", route,
    {"search":"search", "ingest":"ingest",
     "rag":"rag", "report":"report", "FINISH":END})

# Search/Ingest 完成后回到 supervisor 继续调度
builder.add_edge("search", "supervisor")
builder.add_edge("ingest", "supervisor")

# RAG/Report 生成最终内容后直接结束
builder.add_edge("rag",    END)
builder.add_edge("report", END)
```

**为什么 RAG/Report 直接到 END，而 Search/Ingest 回 Supervisor？**
Search 和 Ingest 是中间步骤，结果还需要 Supervisor 判断下一步；RAG 和 Report 输出的是面向用户的最终答案，无需再次路由。

## 三、完整执行流程示例

以 "帮我搜索 ViT 论文并分析其创新点" 为例：

```
Step 1: Supervisor（首次）
  输入: user_query="帮我搜索 ViT 论文并分析其创新点"
        search_attempted=False, ingested_count=0
  决策: next="search"（用户明确要搜索，未搜索过）

Step 2: Search Agent
  意图识别: "ViT" → arxiv_id="2010.11929"（内置著名论文映射）
  搜索: OpenAlex 精确查询 get_paper_by_id("2010.11929")
  输出: PaperMeta(title="An Image is Worth 16x16 Words", ...)
  更新 state: search_results=[paper], papers_to_ingest=[paper],
              search_attempted=True
  → 自动回到 Supervisor

Step 3: Supervisor（第二次）
  输入: search_attempted=True, pending_count=1
  决策: next="ingest"（有待入库论文）

Step 4: Ingest Agent
  下载: https://arxiv.org/pdf/2010.11929
  解析: PyMuPDF → 全文文本
  分块: RecursiveCharacterTextSplitter(chunk_size=500, overlap=50)
  入库: Chroma.add_documents(docs, ids=["2010.11929__chunk_0", ...])
  更新 state: ingested_count=1, papers[0].ingested=True
  → 自动回到 Supervisor

Step 5: Supervisor（第三次）
  输入: pending_count=0, ingested_count=1, 用户问了技术问题
  决策: next="rag"

Step 6: RAG Agent → END
  检索: "ViT 创新点" → Chroma 召回 top-5 相关段落
  生成: DeepSeek LLM 结合段落生成回答，标注来源
  更新 state: rag_answer="ViT 首次将 Transformer 应用于图像分类..."

Step 7: Supervisor（第四次）
  输入: rag_answer 非空
  决策: FINISH（硬编码：rag_answer 非空直接结束）

Step 8: 后台线程
  异步提取本轮对话偏好 → 更新 SQLite user_profile 表
```

## 四、核心组件详解

### 4.1 AgentState（共享状态）

所有 Agent 读写同一个 state，LangGraph 负责状态流转：

```python
class AgentState(dict):
    messages:         Annotated[list[BaseMessage], operator.add]  # 追加语义
    user_query:       str        # 用户原始输入
    next:             str        # Supervisor 决策的下一个节点
    search_results:   list[PaperMeta]   # 搜索结果
    papers_to_ingest: list[PaperMeta]   # 待入库
    ingested_count:   int        # 已入库数量（从向量库实时读取）
    rag_answer:       str        # RAG 回答（非空→触发 FINISH）
    final_report:     str        # 综述报告（非空→触发 FINISH）
    error:            str        # 错误信息
    search_attempted: bool       # 防死循环：搜索只执行一次
    failed_papers:    list[PaperMeta]   # 失败论文（最多重试2次）
    direct_answer:    str        # Supervisor 的闲聊/引导回复
    thread_id:        str        # 会话 ID，隔离进度队列
```

**messages 为什么用 `operator.add`？**
LangGraph 对 `Annotated[list, operator.add]` 类型的字段执行追加而非覆盖，这样多个节点都能往 messages 里写内容，形成完整的对话历史链。

### 4.2 PaperMeta（论文元数据）

```python
@dataclass
class PaperMeta:
    title:       str
    authors:     list[str]
    abstract:    str
    arxiv_id:    str
    pdf_url:     str
    published:   str
    ingested:    bool = False   # 处理完（成功/失败）后标记 True，防重试死循环
    retry_count: int  = 0       # 失败重试次数，超过 MAX_RETRY=2 永久放弃
```

### 4.3 Supervisor Agent（调度中枢）

**决策优先级**（硬编码 + LLM 决策双保险）：

```
硬编码优先级（代码里直接 return，不走 LLM）：
  1. rag_answer 非空 → FINISH
  2. final_report 非空 → FINISH
  3. error 非空且 ingested_count==0（致命错误）→ FINISH
  4. error 非空且 ingested_count>0（非致命）→ 继续调度（部分失败不影响已有内容）

LLM 决策（经过 prompt 约束）：
  5. search_attempted=True → 绝不再路由到 search
  6. 用户明确要重试 + failed_count>0 → ingest
  7. 用户要搜索 + search_attempted=False → search
  8. pending_count>0 → ingest
  9. ingested_count>0 + 技术问题 → rag
  10. ingested_count>0 + 需要综述 → report
  11. 闲聊/打招呼 → FINISH + reply 字段返回友好回复
```

**为什么要硬编码部分优先级？**
防止 LLM 在边界情况下做出错误决策，比如 rag_answer 已生成但 LLM 还想再次调用 rag；或者 error 轻微时 LLM 直接 FINISH 而丢失已入库的内容。

**闲聊兜底机制**：
LLM 有时把回复内容写进 `reason` 而非 `reply` 字段。代码检测：若 next=FINISH 且 direct_answer 为空、reason 超过15字且不含路由语气词（"完成/已/执行"等）→ 把 reason 提升为 direct_answer。

### 4.4 Search Agent（论文搜索）

**意图识别 → 搜索策略 → 结果解析** 三阶段：

#### 意图识别
```
情况A: 特定著名论文（"首次提出ViT"）→ 内置映射 → arxiv_id="2010.11929"
情况B: 大致标题 → 提取英文标题 → title 精确搜索
情况C: 研究方向（"diffusion model 相关"）→ 关键词泛搜
情况D: arxiv 链接 → 提取 ID
情况E: 追问（"还有吗/其他的"）→ 从对话上下文提取核心话题
         错误: "deep learning"（太宽泛）
         正确: "instance segmentation methods"（基于上下文）
```

**prompt 安全性**：意图识别 prompt 用 `%` 格式化而非 f-string，防止 `chat_context` 里的 JSON 内容（如 `{"type":...}`）触发 `KeyError`。

#### 搜索策略（OpenAlex 优先）
```
精确查询（arxiv_id/title）：
  → OpenAlex 精确接口（稳定，毫秒级响应）
  → 找不到才用 ArXiv MCP 补充（对新论文/冷门论文更准）

泛搜（keyword）：
  → 直接走 OpenAlex，完全规避 ArXiv 限流（"Please wait 60 seconds"）
```

**为什么从 ArXiv 优先改成 OpenAlex 优先？**
ArXiv MCP 频繁限流，每次需要等待 60 秒，严重影响用户体验。OpenAlex 是开放免费 API，覆盖绝大多数 arxiv 论文且有稳定的 arxiv_id 字段，用来下载 PDF 同样有效。

### 4.5 Ingest Agent（论文入库）

**处理流程**：
```
1. 优先处理本地 PDF（前端拖拽上传的临时路径）
2. 过滤掉已标记 ingested=True 的论文
3. 逐篇处理：下载 → 解析 → 分块 → 向量化 → 入库
4. 收集可重试的失败论文（只有有 arxiv_id 的才重试）
```

**去重机制**（三层保障）：
```
层1: 入库前查重（arxiv_id 精确匹配 or 标题匹配）→ 已存在直接跳过
层2: 确定性 chunk ID（"arxiv_id__chunk_N"）→ 相同论文重复写入只覆盖，不产生垃圾数据
层3: ingested 标记 → 本轮流程内不重复处理
```

**付费墙论文处理**：
```python
# 无 arxiv_id → 付费墙（MDPI/Cell等出版商），重试无意义，直接放弃
if not p.arxiv_id:
    continue   # 不加入 retriable_failed

# 有 arxiv_id → 临时网络问题，允许重试最多2次
p.retry_count += 1
if p.retry_count <= MAX_RETRY:
    retriable_failed.append(p)
```

**为什么 has_error 时不直接 FINISH？**
部分论文入库失败不是致命的——已成功入库的论文仍然可以用于 RAG 问答。只有 `ingested_count==0` 且没有任何可用内容时才算致命错误。

### 4.6 RAG Agent（语义问答）

```python
def rag_node(state):
    # 1. 短期记忆注入（理解指代词）
    chat_context = build_context_str(state["messages"])
    
    # 2. 向量检索
    chunks = retrieve_context.invoke({"query": user_query, "k": 5})
    
    # 3. 构造 prompt（含检索段落 + 对话上下文 + 用户问题）
    prompt = RAG_PROMPT.format(
        rag_context=formatted_context,
        chat_context=chat_context or "（无历史记录）",
        user_query=user_query
    )
    
    # 4. LLM 生成带引用的回答
    answer = llm.invoke([HumanMessage(content=prompt)])
```

**与 Report 的区别**：
- RAG：针对具体问题，top-5 段落，温度 0.3
- Report：全量论文列表 + top-8 段落 + 结构化综述模板，温度 0.4（需要一定创造性）

### 4.7 记忆机制

#### 短期记忆（context_utils.py）
```python
def build_context_str(messages: list, window=4) -> str:
    """
    从 messages 里提取最近 4 轮有效对话（过滤路由日志）
    返回格式：
        用户：帮我搜索 ViT 的论文
        助手：已找到并入库...
        用户：它和 CNN 相比...  ← "它"在下一轮可被正确理解
    """
```

注入位置：Supervisor / Search / RAG / Report 所有 Agent 的 prompt。

**为什么不直接传完整 messages？**
messages 里混杂了大量 Supervisor 路由日志和 Ingest 进度消息，噪音太多，浪费 token；滑动窗口 + 过滤只保留面向用户的实质内容，约 800 字以内。

#### 长期记忆（user_memory.py）
```python
# 每次对话结束后，后台线程异步执行：
def extract_and_save_profile(messages, llm_model, ...):
    # 1. 过滤路由日志，提取面向用户的对话对
    # 2. 调用 LLM 分析提取：研究方向、常搜话题、关注来源
    # 3. 合并到已有偏好（新数据优先，保留最多10条，去重）
    # 4. 写入 SQLite user_profile 表（UPSERT）

# 新会话 Supervisor 调用：
profile = load_user_profile()
profile_str = format_profile_for_prompt(profile)
# 格式示例：
# - 研究方向：目标检测、实例分割
# - 常搜话题：open-vocabulary detection, CLIP-based methods
# - 上次会话关注：Mask2Former, SAM
```

**合并策略**：research_interests / frequent_queries / preferred_venues 用去重追加（保留历史 + 补充新内容）；last_session_topics 每次直接覆盖（反映最新关注点）。

### 4.8 MCP 协议与工具调用

**MCP（Model Context Protocol）** 是 Anthropic 提出的标准化工具调用协议，统一了不同来源工具的调用方式。

**本项目的两个 MCP Server**：

| Server | 传输方式 | 角色 | 特点 |
|--------|----------|------|------|
| 本地 OpenAlex MCP | stdio（子进程） | 主要搜索后端 | 稳定，无限流，FastMCP 实现 |
| 远程 ArXiv MCP | streamable_http | 精确查询补充 | ModelScope 部署，偶有限流 |

**工具列表缓存（性能优化）**：
```python
_tools_cache: dict[str | None, list] = {}

async def _get_tools_cached(server_name):
    if server_name in _tools_cache:
        return _tools_cache[server_name]  # 直接复用
    client = MultiServerMCPClient(_MCP_CONFIG)
    tools = await client.get_tools(server_name=server_name)
    _tools_cache[server_name] = tools    # 缓存
    return tools
```

异常时自动 `invalidate_tools_cache`，下次重新建立连接，做到复用与容错平衡。

**同步包装**：
```python
def call_mcp_tool(tool_name, tool_input, server_name=None):
    """同步包装器，供各 Agent 直接调用（Agent 本身是同步函数）"""
    return asyncio.run(_call_tool_async(tool_name, tool_input, server_name))
```

**OpenAlex get_paper_by_id 的六种查找方式**：
```
1. arxiv:ID 格式 → OpenAlex 内部 ID
2. DOI 格式（10.48550/arXiv.xxxx）→ 适合2022年后论文
3. https URL 过滤 → locations.landing_page_url
4. http URL 过滤 → 老论文常见
5. 不带协议的 URL 过滤
6. arxiv_id 作为关键词搜索兜底（验证结果）
```

### 4.9 SSE 流式进度推送

```
后端执行流程（同步线程）          前端（异步 SSE）
    ↓                              ↓
各 Agent push_progress()  →  进度队列  →  drain_queue()  →  "think"事件
    ↓                                                           ↓
  状态快照                                               前端思考面板追加一行
    ↓
stream_mode="values" 快照 →  节点变化推断  →  "step"事件（🔍搜索/📚入库/💬问答）
    ↓
工作流结束 → "done"事件（含最终答案）
```

**为什么用 `stream_mode="values"` 而非默认的 `"updates"`？**
`updates` 模式只返回节点**修改**的字段（增量），`values` 模式返回执行该节点后的**完整 state 快照**。用 `values` 可以直接拿最后一个快照作为终态，不需要在前端拼合增量，避免了字段丢失问题（尤其是 `direct_answer` 在闲聊场景下的丢失）。

**done 事件的 type 字段冲突问题**：
SSE 协议用 `type` 区分事件种类（step/think/error/done），但 `_extract_answer` 返回的 `type` 是消息渲染类型（rag/info/ingest 等），两者撞字段名。解法：done 事件用 `msg_type` 承载渲染类型，`type` 只作 SSE 协议标识。

### 4.10 多轮对话持久化

```python
# 每个 thread_id 对应独立的 checkpoint
config = {"configurable": {"thread_id": thread_id}, "recursion_limit": 20}

# 续接对话时的 input_state：
input_state = {
    "user_query": new_query,
    "next": "supervisor",
    "search_attempted": False,  # 每轮重置
    "rag_answer": "",           # 每轮重置
    # ← 不传 messages（LangGraph 从 checkpoint 自动 merge）
    # ← 不传 ingested_count（实时从向量库读取，避免跨 session 计数偏低）
    "ingested_count": len(list_ingested_papers()),  # 实时查库
    "failed_papers": prev_failed,  # 保留上轮失败论文
}
```

**为什么 ingested_count 不从 checkpoint 读取？**
多个会话可能共用同一个向量库，如果 session A 入库了论文，session B 续接时从 checkpoint 读到的 ingested_count 是 0（因为 B 自己没入过库），Supervisor 会误判向量库为空而走搜索流程。实时读向量库确保数值准确。

## 五、设计亮点与权衡

### 5.1 防死循环三重保护

| 保护层 | 机制 | 防止什么 |
|--------|------|----------|
| `search_attempted` | 搜索完标记 True，Supervisor 不再路由 search | 搜索失败无限重试 |
| `ingested=True` | 成功/失败都标记，过滤掉已处理论文 | ingest 循环处理同一篇 |
| `retry_count <= MAX_RETRY` | 失败计数，无 arxiv_id 直接放弃 | 付费墙论文无限重试 |

### 5.2 Supervisor 的双保险决策

代码硬编码（速度快、确定性高）+ LLM 决策（灵活处理复杂意图）结合：

- **硬编码**负责处理明确状态（rag_answer 非空必须 FINISH，search_attempted=True 不能再搜索）
- **LLM**负责理解模糊意图（"帮我找更多这方向的论文" → search，还是 "还有什么区别" → rag）

### 5.3 MCP vs 直接 API 调用

为什么用 MCP 协议而不是直接调用 OpenAlex HTTP API？
- **标准化**：不同搜索后端（ArXiv/OpenAlex/Semantic Scholar）有统一的调用接口
- **可插拔**：新增搜索源只需写一个 MCP Server，不改 Agent 代码
- **进程隔离**：本地 OpenAlex MCP 跑在子进程里，崩溃不影响主进程

### 5.4 向量库实时查询 vs 状态计数

`ingested_count` 在续接会话时实时查向量库（`list_ingested_papers()`），而不是用 checkpoint 里存的数字。多了一次 IO，但确保了跨会话一致性，避免 Supervisor 误判。

## 六、可能被问到的问题与回答

### 架构类

**Q: 为什么选 LangGraph 而不是 LangChain LCEL 或直接写状态机？**
A: LangGraph 的 StateGraph 天然支持有向图（循环路由）、状态持久化（Checkpointer）和流式执行（stream_mode）。LCEL 是线性链式，不支持 Supervisor 这种"执行后回到中枢再决策"的循环模式。直接写状态机需要手动管理状态传递、持久化、流式推送，工程量大且不标准化。

**Q: AgentState 为什么用 dict 子类而不是 Pydantic BaseModel？**
A: LangGraph 的 StateGraph 要求状态是可以按字段 merge 的对象。dict 子类配合 `Annotated` 类型注解可以自定义 merge 策略（如 messages 的 `operator.add` 追加语义）。Pydantic 也可以，但需要额外配置 reducer 函数，不如 Annotated 简洁。

**Q: Supervisor 用 LLM 做路由决策会不会太慢/太贵？**
A: 有权衡。优点是能理解自然语言意图（"还有吗"、"换个角度"等模糊指令）；缺点是多了一次 LLM 调用。缓解措施：temperature=0 让决策确定，减少输出 token（只输出 JSON），硬编码处理明确状态减少 LLM 调用次数。未来可以用分类模型替代 LLM 做路由。

### RAG 类

**Q: chunk_size=500 是怎么定的？chunk_overlap=50 有什么用？**
A: 500 字约等于 1-2 个自然段，语义完整且不太长（避免向量化质量下降）。50 字重叠确保段落边界处的信息不被截断——比如一句话跨越两个 chunk 时，两个 chunk 都能检索到相关信息。

**Q: 为什么用 Chroma 而不是 FAISS 或 Pinecone？**
A: Chroma 支持本地持久化（不需要额外服务），支持按 metadata 过滤（`where={"arxiv_id": "xxx"}`），且有官方 LangChain 集成。FAISS 只是纯向量索引，没有 metadata 存储；Pinecone 是云服务，有网络依赖。对于本地单用户场景，Chroma 最合适。

**Q: 如果向量库里有很多论文，检索准确率会下降吗？**
A: 会有影响，但 top-k 检索（k=5）在实践中对大多数问题足够。可以优化的方向：增加 metadata 过滤（先按论文 title/date 过滤再检索）、使用 HyDE（先让 LLM 生成假设答案再检索）、或 reranker 对 top-k 结果重排序。

### 记忆类

**Q: 短期记忆为什么选 4 轮窗口？**
A: 权衡 token 消耗和上下文质量。每轮对话约 200-400 字，4 轮约 800-1600 字，加上 rag_context 和 prompt 本身，总长度可控。实验发现超过 6 轮后，更早的对话内容对当前问答的帮助已经很小，但 token 消耗线性增长。

**Q: 长期记忆是如何避免偏好"污染"的？**
A: 几个设计：① 只提取明确出现的信息，不推断（prompt 里明确要求）；② last_session_topics 每次覆盖，只反映最近关注；③ 合并时新数据优先、最多保留10条，自然淘汰过时偏好；④ 提供 DELETE 接口允许用户主动重置。

### 工程类

**Q: SSE 和 WebSocket 有什么区别？为什么选 SSE？**
A: SSE 是单向的（服务器→客户端），基于 HTTP，天然支持断线重连，前端用标准 `EventSource` API；WebSocket 是双向的，需要握手升级，适合实时交互场景。本项目是"用户发一次请求，服务器流式返回进度"的模式，单向足够，SSE 更简单且不需要额外库。

**Q: 为什么用线程而不是协程来做进度推送？**
A: LangGraph 的图执行是同步的（每个节点是普通函数），无法直接 await。进度队列（`queue.Queue`）是线程安全的，在同步的 Agent 节点里 `push_progress()` 写入，在异步的 FastAPI SSE 生成器里 `drain_queue()` 读取，用 `asyncio.to_thread` 桥接同步执行和异步 SSE 生成器。

**Q: PaperMeta 反序列化警告是什么意思？**
A: LangGraph 的 SQLite Checkpointer 用 MessagePack 序列化 state。`PaperMeta` 是自定义 dataclass，不在 LangGraph 的默认允许类型列表里，反序列化时会有警告。不影响功能，可以通过设置 `LANGGRAPH_STRICT_MSGPACK=false` 消除，或把 `PaperMeta` 添加到 `allowed_msgpack_modules`。

## 八、项目亮点总结（一句话版）

1. **LangGraph Supervisor 模式**：多 Agent 协作，状态共享，条件路由，防死循环
2. **OpenAlex 优先搜索策略**：规避 ArXiv 限流，精确查询时再用 ArXiv 补充
3. **付费墙论文自动识别**：无 arxiv_id 的论文直接放弃重试，防止死循环
4. **双层记忆机制**：短期滑动窗口（4轮）理解指代词 + 长期 SQLite 跨会话用户画像
5. **SSE 实时进度**：`stream_mode="values"` 完整快照 + 线程安全进度队列，前端思考面板
6. **MCP 工具缓存**：首次连接后工具列表复用，异常时 invalidate 重连
7. **幂等入库**：确定性 chunk ID（`arxiv_id__chunk_N`），重复写入只覆盖不产生垃圾
8. **f-string 安全**：对话上下文可能含 JSON 字符串，所有 prompt 拼接用 `%` 格式化
